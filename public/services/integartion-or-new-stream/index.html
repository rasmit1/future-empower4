Integration or New Product Stream

The fork in the road. Do we embed AI into existing workstreams or create an entirely new AI-powered product stream? This decision shapes everything that follows.

“The integration decision is not a technology decision. It is a change management decision wearing a technology costume.”

The burning question this phase answers#

How does this AI product reach users? Do we integrate it into what already exists — or build a new product stream around it?

There is no universally right answer. The right answer depends on the nature of the AI capability, the maturity of existing workflows, and the appetite for change in the organization.

Why this situation arises#

This is the phase most organisations have no framework for. They have built the product, proved it works, got it production-ready — and then face a question nobody prepared them for. Where does this actually go?

The wrong decision here does not kill the product — but it dramatically slows adoption and inflates cost. Integrating into an existing system when the AI capability demands a new workflow forces users into a broken experience. Building a standalone new stream when simple integration would suffice creates unnecessary complexity.

The two paths#

Path A — Integrate into existing workstreams#

AI capability is embedded into existing tools and workflows. Users do not need to change where they work — AI comes to them.

Best when: AI augments existing tasks without fundamentally changing the workflow.

Strengths

  • Lower adoption friction — familiar environment
  • Faster time to value — no new system to learn
  • Lower change management investment

Risks

  • Existing system constraints may limit what the AI can do
  • Technical debt in existing systems can complicate integration
  • May cap the ceiling of what the AI can deliver

Path B — New AI-powered product stream#

AI capability powers a new product, workflow, or service category. Users adopt a new way of working.

Best when: AI fundamentally changes how the work is done — not just augments it.

Strengths

  • Higher upside — potential to create entirely new value
  • No legacy constraints limiting the AI capability
  • Positions AI as a core product not a feature

Risks

  • Higher investment and longer adoption curve
  • More change management required
  • More complexity to sustain long term

The hybrid trap#

Many organisations try to do both simultaneously — integrate into existing systems AND build a new product stream.

This almost always produces the worst of both worlds.

Partial integration that confuses users and a new stream that never gets the investment it needs to succeed. Choose one path clearly. You can add the second later from a position of strength.

How to make the decision — five questions#

Question Integrate New stream
Does the AI change how users work or just what they access? Just access Changes how
Can the existing system support what the AI needs technically? Yes No — too constrained
How much workflow change are users ready for? Low appetite High appetite + strong sponsor
What is the time to value expectation? Short timeline Longer horizon acceptable
Is this AI capability a core product or a feature? Feature Core product

Score your answers. More ticks in the left column — integrate. More in the right — new stream.

Recognise your situation#

Use case 01 — The integration that broke the user experience#

Problem: AI was integrated into an existing system but the user experience became worse not better.

Situation: Existing system architecture could not cleanly accommodate the AI output. Users now have more steps not fewer.

Symptoms:

  • Adoption lower than the standalone POC
  • “The old way was easier” — users reverting
  • IT team spending more time on integration issues than AI improvements

Solution direction:

  • Reassess the integration decision — was this the right path?
  • Consider a lightweight standalone interface alongside the integration
  • Involve users in redesigning the integrated experience

Use case 02 — The new product stream nobody adopted#

Problem: New AI product stream was built but users stayed on the old system.

Situation: New stream required behaviour change that users were not ready for. Change management was underinvested relative to the technical build.

Symptoms:

  • Two systems running in parallel — the old one and the new one
  • New stream used only by early adopters
  • Leadership asking why ROI has not materialised

Solution direction:

  • Invest in launch and adoption with the same intensity as the build
  • Define a sunset date for the old system to force migration
  • Involve reluctant users in co-designing the new stream experience

Team level vs enterprise level#

Team level#

  • Decision made with input from both technology and user-facing teams
  • User journey mapped for both paths before the decision is made
  • Change management plan created alongside the technical integration plan
  • Pilot group identified — integration tested with real users before full rollout

Enterprise level#

  • Integration decision governed at portfolio level — consistency across AI products
  • Enterprise architecture review — how does this affect the broader technology landscape
  • Data and API standards defined — so integrations are maintainable not bespoke
  • Sunset plan for legacy systems — if a new stream is chosen what happens to what it replaces

Phase completion checklist#

Before moving to next Phase confirm all of the following:

  • Decision made — integrate or new product stream — one path chosen clearly by the right decision maker
  • User journey mapped — the end-to-end experience for the user is designed and validated — not just the technical integration
  • Change management plan created — how users will be moved to the new experience is planned with the same rigour as the technical build
  • Pilot group identified — a small group of real users will test the integration before full rollout
  • Architecture sign-off — the integration approach has been reviewed and approved by the right technical stakeholders
  • Rollout plan sequenced — who gets access first, second, and last — and why

The non-obvious insight#

The question is not what is technically possible — it is what users will actually adopt.

The most elegant integration that users ignore is worth nothing. The messier new stream that users embrace is worth everything.

Adoption is the only metric that matters in this phase. Build your decision around it.
